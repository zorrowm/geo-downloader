## #2 清理临时目录前瓦片是否已入缓存 db

- **问题**：`cleanup_temp_dir` 删除临时瓦片前，这些瓦片是否都已写入缓存 db？会不会丢数据？
- **结论**：**是，不会丢数据。**
- **依据**（`src-tauri/src/downloader.rs` 约 L375-460）：
  - 下载阶段每张瓦片的 async future 内，网络下载成功且启用缓存时即 `tcache::Store::global().put(...)` 写入缓存 db。
  - `cleanup_temp_dir` 仅在全部下载 + 合并/导出完成后才执行，时序上晚于所有缓存写入。
- **例外**（这些情况瓦片不进缓存，但属预期行为，非数据丢失）：
  - 缓存功能关闭；
  - 瓦片 >4MB；
  - 空字节；
  - 单张 `put` 失败（会计数 + 记日志，但该瓦片仍用于本次导出）。

## #3 Wayback 不同版本「都是同一期」

- **现象**：群友反馈用 Wayback 下载不同历史版本，下载下来发现都是同一期影像。
- **结论**：**非程序 bug，是 Esri Wayback 数据源固有特性。**
- **代码静态核查**（全链路按版本区分，均正确）：
  - 前端单/批量模式分别把各自 `version_id`（`v.id`）传入 `createWaybackTask`，批量每个版本写不同文件名（含日期+时间戳）。
  - 后端 `make_tile_source(version_id)` 把 `version_id` 拼进瓦片 URL 路径 `.../tile/{version_id}/{z}/{y}/{x}`，且 `source.id = wayback_{version_id}`。
  - 缓存 key `SourceKey(wayback_{version_id})` 按版本区分，无碰撞。
- **实证证据**（经代理直连 Esri 瓦片服务，比对同一瓦片坐标不同版本的字节 MD5）：
  - 上海陆家嘴 z16：版本 10842/22869/22252/13192（2026-05 ~ 2025-12）**字节完全相同**（同 MD5 `7073FC...` 同大小 20682）。
  - 雄安新区 z16（2017 后剧变区）：版本 10(2014-02)=31144(2014-06) `F56C...`（两个 2014 版本完全相同）≠ 13192(2025-12) `DBCF...` ≠ 10842(2026-05) `72004...`（三组哈希各异）。
  - 即：**服务器确实按 version_id 返回影像**（跨年版本字节不同），但**同一地点相邻多个 release 常对应同一次底图采集**，故返回字节完全相同的瓦片。
- **根因**：Wayback 时间轴列出的是**全球范围 release**；任意具体位置的影像只在少数日期真正更新。批量下载该位置的多个相邻 release，会得到同一期采集的相同影像，并非程序取错版本。
- **建议给用户**：使用「**增量**」模式——它通过逐位置元数据扫描（`scan_wayback_metadata`），只识别本地区域影像**真正变化过**的那些 release，避免重复下载同一期。
- **处置**：无需改动。可考虑在批量/单版本模式 UI 增加提示文案，引导用户用增量模式（非必须，待定）。
